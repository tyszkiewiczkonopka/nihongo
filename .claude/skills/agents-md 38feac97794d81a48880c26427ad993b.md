# agents-md — Generate Repository Guidelines for AI Agents

Produce an [AGENTS.md](http://AGENTS.md) that onboards AI coding agents to this repository. The file is short, repo-specific, and structured so the most important rules appear first. Nothing generic — every line must trace back to something you actually found in the codebase.

## Input

Optional argument:

- empty → write [AGENTS.md](http://AGENTS.md) at the repo root
- directory path → write [AGENTS.md](http://AGENTS.md) inside that directory (local guide)
- full .md path → write there verbatim

If the target file already exists, switch to the Update flow below.

## Scope

**Repo-level** (target = repo root): high-level onboarding — structure, commands, conventions, CI gate, commit style.

**Directory-level** (target = subdirectory): local rules only — what files in this directory look like, what's forbidden here, how to add a new unit following the existing pattern. 120–250 words. No repo-wide overview.

## Create flow

1. **Discover** — read in order, skip what doesn't exist: [README.md](http://README.md), existing [AGENTS.md](http://AGENTS.md), docs/ index; manifest (pom.xml, build.gradle, package.json, pyproject.toml, etc.); lint/format/type configs; test config and a sample test file; CI (.github/workflows/ — one or two files, enough to know the gate); directory layout (top two levels); history (git log --oneline -n 30 for commit conventions)
2. **Extract** — write down for yourself: the 3–5 commands an agent runs most often (build, test, lint, run); conventions a reviewer would actually flag in PR (naming, file placement); any hard "never do X" rule in README, CI validators, or existing docs; where deeper docs live (to reference, not duplicate)
3. **Draft** — per Output structure below.
4. **Quality check** — run all five guards before writing. Revise if any fail.
5. **Write** — single file write to the resolved path. Confirm path and word count.

## Update flow

Triggered when the target file already exists.

1. Read the existing file fully.
2. Run git log --follow --format="%h %ad" --date=short -- <path> to find LAST_TOUCH (last commit that touched the file).
3. Diff repo state since LAST_TOUCH: manifest changes (scripts, deps, engines); lint/test config changes; CI workflow changes (--stat); check every path the file references still exists
4. Classify each line: KEEP / UPDATE / REMOVE / MISSING
5. Ask: "Apply updates / Show me the list first / Full regenerate / Cancel"
6. Edit surgically — prefer targeted edits over a full rewrite
7. Re-run quality guards on the result

## Output structure

Title: # Repository Guidelines

Target length: 200–400 words of body.

Section order (adapt to what the repo actually has — omit empty sections):

1. Hard rules / Never do X — tripwires first, if any exist
2. Project structure — top-level map, where source/tests/assets live
3. Build, test, and run commands — 3–6 commands with one-line purpose each
4. Naming and coding conventions — actual patterns, not generic advice
5. Testing — framework, where tests live, how to run a single test
6. Commits and PRs — convention from git log, required CI checks
7. Architecture notes (optional) — 3–5 bullets or a reference link

Open with 1–2 sentences: what the project is and its primary stack.

## Quality guards

All five must pass before writing:

1. **Length** — 200–400 words of body. Under 200 = too vague. Over 400 = padded.
2. **No multi-line snippets** — use @path/to/file references instead of inlining code or config blocks. Single-line commands are fine.
3. **Every rule is checkable** — re-read each sentence: could a reviewer flag a diff against this? Strike "write clean code", "follow best practices", "handle errors properly" — rewrite with a concrete pattern or tool name.
4. **No redundant knowledge** — strike anything true of all Spring Boot projects, all Java projects, or all software projects. Only project-specific knowledge earns a line. Replace duplicated README content with @[README.md](http://README.md).
5. **Critical rules first** — "never do X" and most-used commands in the first third of the file. If the only hard rule is at the bottom, move it up.

## What this skill does NOT do

- Does not invent facts. Every claim traces back to a file or command inspected.
- Does not write generic advice ("write clean code", "follow best practices").
- Does not repeat framework defaults the agent already knows from training.
- Does not touch any file other than the target [AGENTS.md](http://AGENTS.md).
