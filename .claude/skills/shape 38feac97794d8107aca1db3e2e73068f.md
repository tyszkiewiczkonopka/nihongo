# shape — Structured Discovery Before Writing a PRD

Walk from "I have an idea" (greenfield) or "I want to change this system" (brownfield) to a structured [shape-notes.md](http://shape-notes.md) that the prd skill can turn into a PRD.

This skill is a facilitator, not a generator. It never writes vision, requirements, or business logic the user didn't say. Its value is the questions and their order — not the answers it offers.

## When to use vs research+frame

| Situation | Use |
| --- | --- |
| Bug fix, refactor, new endpoint in existing service | research → frame → plan |
| New significant feature in an existing service | shape → prd → plan |
| New microservice | shape → prd → plan |
| Major architectural change or new module | shape → prd → plan |

If in doubt: shape is appropriate when the work warrants a PRD. research+frame is appropriate when it doesn't.

## Initial response

If a seed idea was provided inline, capture it verbatim and proceed to Step 0.

If nothing was provided, ask the person to share the seed idea (what they want to build or change, in their own words) and optionally any rough notes, sketches, or files to read. Then wait.

## Step 0: Detect context type

Determine whether this is greenfield (new) or brownfield (existing system). Check for git history and lockfile/manifest presence (package-lock.json, pom.xml, build.gradle, etc.). Git history + lockfile/manifest → brownfield. No signals → greenfield. Confirm with the user before continuing.

## Steps 1–6: Discovery

Each step follows the same pattern: open with a single question to elicit the user's first attempt; echo back what you heard distinctly, not collapsed; surface 2–4 ambiguities as concrete options with tradeoffs; lock the decision back to the user before writing anything to disk; write the section to docs/project/[shape-notes.md](http://shape-notes.md).

Hard rules across all steps: never generate content the user didn't say; never commit to a tech stack, framework, or database (those come after the PRD); if a section needs a value the user hasn't provided, ask, don't invent; missing information goes to Open Questions, not filled with guesses.

### Step 1: Vision & problem

Greenfield: ask who has the pain, what moment they feel it, and what it costs them today. Echo back Pain / Person / Moment / Cost today as four separate bullets. Challenge vague answers ("everyone", "always") by asking who specifically they've seen experience this in the last month.

Brownfield: ask what exists today, who uses it, and what pain or missing capability is driving this change. Echo back Current system / Tech stack mentioned / Users / Pain or gap / Must preserve as five separate bullets. Challenge "must preserve" by asking what would page them if this change broke something tomorrow.

Write: Vision & Problem Statement + User & Persona sections. Brownfield also writes a Current System section.

### Step 2: Access control

Greenfield: ask how the user gets into the service (auth, API key, internal-only, no auth). Brownfield: ask if the auth model is changing as part of this work, or if new roles are being added. If auth isn't changing, record the current model with a note that no changes are planned.

Write: Access Control section.

### Step 3: MVP scope and timeline

Greenfield: ask to describe the smallest end-to-end flow that would prove this works, step by step. Echo back as a numbered sequence. If the flow has more than ~6 steps before producing value, or requires multiple integrations before any user-visible payoff, surface the cost explicitly and offer two paths: scope down to the smallest slice that proves the concept, or commit to the larger scope with eyes open on the effort.

Brownfield: ask to describe the smallest incremental change that proves this improvement works, and what a user does differently after this ships. Also ask about blast radius — which existing features or integrations could break.

Write: Success Criteria section (Primary / Secondary / Guardrails).

### Step 4: Functional requirements

Ask what the system has to be able to do, based on the flow described. Format each requirement as: FR-NNN: [Actor] can [capability]. Priority: must-have or nice-to-have.

Brownfield: also tag each as Features: new, modified, or preserved. Explicitly prompt for preserved FRs — which existing capabilities must survive this change, since making preservation explicit prevents accidental breakage.

After capturing FRs, run a Socrates round — one challenge per FR, asking what would have to be true for this FR to hurt the product instead of help it. Always include "No counter-argument — it stands as written" as the last option.

Write: Functional Requirements + User Stories sections.

### Step 5: Business logic and constraints

Ask to describe the rule of operation in one sentence — the decision this service/module makes that distinguishes it from a generic CRUD list. If the answer reduces to basic CRUD with no domain decision, flag it explicitly: what you've described is a CRUD list with no domain decision; a real domain rule answers what the system decides FOR the user (examples: recommendation, prioritisation, classification, validation, scoring, workflow state transitions, calculation); ask what rule their service applies.

Brownfield: ask what the existing domain rule is, and whether this change adds a new rule, modifies the existing one, or is infrastructure-only.

Write: Business Logic + Non-Functional Requirements sections. Brownfield also writes Constraints & Preserved Behavior.

### Step 6: Product framing and non-goals

Ask three short questions one at a time: what kind of thing is being built (web app, API/backend service, CLI, mobile app, library, data pipeline); roughly how many consumers/users (internal only, dozens to a hundred, up to ten thousand, more than ten thousand); is there a hard deadline, and is this after-hours work or a day job.

Then one multi-select round: what is this explicitly NOT doing. Generate 3–5 options specific to this project's domain, not generic. Always include "Other" for free text.

Write: Non-Goals section + update frontmatter with product_type, target_scale, timeline_budget.

### Step 7: Quality cross-check

Read back the full [shape-notes.md](http://shape-notes.md) and check: Access Control is non-empty and non-placeholder; Business Logic is one declarative sentence (not "TBD"); timeline is acknowledged (either short scope or explicit cost acknowledgement); Non-Goals has at least one entry; for brownfield, preserved behavior is explicitly named.

For each missing element, name it and state the consequence (e.g. "Business Logic not captured — the PRD will be hollow without a domain decision"). Ask: address gaps now / accept and finish / restart a specific phase.

### Step 8: Hand off

Final write of docs/project/[shape-notes.md](http://shape-notes.md). Present a summary: project name, context type, number of FRs drafted, quality check status. Point to the notes file and suggest "Next: prd".

Stop. Do not chain into prd automatically.
