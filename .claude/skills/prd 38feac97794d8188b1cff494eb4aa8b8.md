# prd — Generate a PRD from Shape Notes

Take a shaped notes file and emit a docs/project/[prd.md](http://prd.md) that conforms to the PRD schema. Route every gap to Open Questions rather than inventing content.

This skill is a document generator, not a discovery facilitator. It never invents domain decisions, business logic, success criteria, or user stories. If it's not in the input, it goes to Open Questions.

## When to use

Use when docs/project/[shape-notes.md](http://shape-notes.md) exists with a quality-check block, or when you have raw notes to turn into a PRD draft. Skip when you're still ideating — run shape first.

## Initial response

Default input path: docs/project/[shape-notes.md](http://shape-notes.md). If a different path was provided as argument, use that. If the file doesn't exist, offer to run shape first (recommended), accept pasted raw notes with a warning about thin input, or cancel.

## Step 1: Determine context type

If context_type is in the input frontmatter, use it directly. If not, auto-detect from cwd using the same logic as shape's Step 0, and confirm with the user.

## Step 2: Assess input quality

Score the input on four signals (1 point each): a checkpoint frontmatter block present (came from shape); at least one FR-NNN format requirement; at least one Given/When/Then user story; an explicit one-sentence business rule in the Business Logic section.

Score ≥ 2: proceed silently. Score < 2: warn with specifics — name each missing signal and its consequence for the generated PRD. Ask: run shape first / proceed anyway / cancel.

## Step 3: Generate PRD

Read the input fully. Build the PRD in memory first — do not write to disk yet.

### Frontmatter

Include: project (from input or directory name), version: 1, status: draft, created date, context_type (greenfield or brownfield), product_type, target_scale, timeline_budget. For any missing field, emit a TODO marker and add a matching entry under Open Questions.

Do NOT include: framework names, database names, deployment targets, team composition, language preferences. Those belong downstream of the PRD.

### Sections — greenfield (10 sections, in this exact order)

Vision & Problem Statement; User & Persona; Success Criteria (with Primary/Secondary/Guardrails subsections); User Stories; Functional Requirements; Non-Functional Requirements; Business Logic; Access Control; Non-Goals; Open Questions.

### Sections — brownfield (11 sections, in this exact order)

Current System Overview; Problem Statement & Motivation; User & Persona; Success Criteria (with Primary/Secondary/Guardrails subsections); User Stories; Scope of Change; Constraints & Compatibility; Business Logic Changes; Access Control Changes; Non-Goals; Open Questions.

### Content rules for each section

If the input has matching content: transcribe faithfully, preserving the user's wording. Convert formatting only when the schema requires it (FR-NNN format, Given/When/Then). Do not rephrase, summarise, or "improve" the user's words.

If the input has partial content: transcribe what's there, close with a TODO marker for what's missing, and add a matching Open Questions entry.

If the input has no matching content: emit the heading plus a TODO marker pointing to Open Questions.

Hard rule — never invent business logic. If there is no one-sentence rule in the input, Business Logic must read as a TODO pointing to Open Questions. No exceptions, no extrapolation from entity names or FRs.

### What must NOT appear in the PRD

Scan all sections before writing. Flag and remove: framework, database, or hosting platform names (except in Current System Overview for brownfield); schema/column-level design details; runtime location specifics; enforcement mechanism details; transport/protocol names; UI affordances used as NFRs.

If any of these are in the input, move them to a Forward: technical notes section — do not drop them, do not include them in PRD sections.

## Step 4: Self-review before writing

Check: all required sections present, in order, exact spelling; no forbidden vocabulary in PRD sections; frontmatter has all required keys; Success Criteria has all three subsections (or TODOs). If any check fails, abort the write and report what failed specifically — do not patch silently.

## Step 5: Collision check

If docs/project/[prd.md](http://prd.md) already exists, offer to save as [prd-v2.md](http://prd-v2.md) (recommended, preserves history), overwrite [prd.md](http://prd.md), or abort.

## Step 6: Write and hand off

Write the file. Present a summary: project name, context type, file path, sections populated count, number of Open Questions entries, and a list of which sections have TODOs. Suggest "Next: plan <feature-id>".

Stop. Do not chain into planning automatically — the developer should review the PRD before proceeding.
