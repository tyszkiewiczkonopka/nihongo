# implement — Execute a Phased Implementation Plan

Implement an approved plan from docs/features/<feature-id>/[plan.md](http://plan.md), phase by phase. Each phase ends with automated checks, a manual confirmation gate, and a commit. The Progress section in the plan is the single source of truth for execution state.

## Initial setup

When invoked as "implement <feature-id> [phase N]", resolve to docs/features/<feature-id>/[plan.md](http://plan.md).

If nothing was provided, ask for a feature-id and point to "ls docs/features/" to list active changes. Remind that the plan should be reviewed before starting.

When given a plan:

1. Read the plan completely
2. Read [AGENTS.md](http://AGENTS.md) if present — treat every rule as a prior that shapes implementation choices in this run
3. Read all files referenced in the plan (research, frame, source files)
4. Read files fully — never partially
5. Find the next pending step: scan Progress for the first unchecked item. If a phase N argument was passed, jump to the first unchecked item inside that phase instead

## Implementation philosophy

- Follow the plan's intent while adapting to what you find
- Implement each phase fully before moving to the next
- Update progress checkboxes as you complete each step
- The plan is your guide, but your judgement matters when reality diverges

When the codebase doesn't match the plan, stop and present it explicitly: what was expected, what was found, and why this matters. Then ask: Adapt and continue / Skip this part / Stop and re-plan. The developer decides. Never silently diverge from the plan.

## Tracking touched files

Maintain a set of files modified or created during the current phase. This is the exact input to git add at phase end — never use git add . or git add -A. Always include the [plan.md](http://plan.md) file in this set since the Progress section is modified every phase. Reset the set at each phase boundary.

## Tracking issue references

Before any commit, scan the conversation for tracking references (Jira keys, Linear IDs, GitHub issue numbers). If found, include them in commit messages under a Refs line.

## Phase-end ritual

After all automated checks pass for a phase:

### 1. Manual confirmation gate

Present what automated checks passed and what manual items need verification from the plan. Wait for confirmation — do not proceed until the developer confirms.

### 2. Stage explicitly

Run git add for each file in the touched-file set by name. Never git add . or git add -A.

### 3. Check for unrelated dirty files

Run git status --porcelain. If files outside the touched set are dirty, ask whether to stage only the planned set (recommended), stage all, or abort.

### 4. Commit

Propose a Conventional Commits message in the form: <type>(<feature-id>): <phase title> (p<N>), where type is feat/fix/refactor/chore/test depending on the nature of the change. Include a short body listing touched files and a Refs line if applicable. Ask for approval, edits, or override. Never use --no-verify, --amend, or --force.

### 5. Capture SHA and write back

Get the short commit SHA and append it to every Progress row flipped during this phase.

### 6. Next phase decision

Ask whether to continue to the next phase directly, clear context first (stop and hand off the next command), or review this phase before continuing.

## State tracking

The Progress section in [plan.md](http://plan.md) is the only source of truth. No state files, no comment markers. To find current position, scan for the first unchecked item. Completion = checked / total. Mutate only the Progress section — never edit phase content blocks.

## After all phases

When every item in Progress is checked:

1. Scan once more for any remaining unchecked items — surface stragglers explicitly
2. Update feature.md: set status to implemented
3. Run an epilogue commit to land the SHA write-back and status change
4. Present a completion summary: phases completed, files changed, and suggest a final review or marking as done
